# CLAUDE.md

> Project context file for Claude Code sessions. Read this file first, every session.

## Project overview

noCluNetwork is the noClu estate's cross-platform community-engagement engine. It is a Fastify Core API that runs communities, membership, leveling, and moderation across chat platforms, resolves platform accounts to noClu identities, and acts as the bidirectional bridge between those platforms and noCluID. It is a relying party of noclulabs.com, never an auth or identity issuer.

- Repository: github.com/noclulabs/noCluNetwork
- Role: the community engine, plus the bridge between platform bots and noCluID
- Hosting: Docker on the shared noClu DigitalOcean droplet, co-located with noclulabs.com in the noCluHub VPC. There is no public route today: all bridge traffic is outbound over the private Docker network, and the one inbound endpoint (summon) is disabled and unexposed until noCluBot. A Caddy-fronted public surface on host port 3000 is intended future state, not the current shape. See DEPLOYMENT.md.
- Status: the phase 5 bridge is deployed and live in production as of 2026-06-30 (verify, emit, and reconcile enabled; summon disabled pending noCluBot). ROADMAP.md is authoritative for phase status; CHANGELOG.md holds the full history; DEPLOYMENT.md is the deployment record.

This is the rebuild of the legacy portalNetwork Core API and bot suite. The old code was mined for a spec, not integrated. The legal entity stays portalNetwork Inc.; the product and the repo are noCluNetwork.

## The noClu suite model

noclulabs.com is the estate's auth issuer and identity core (it owns Auth.js, noCluID, and the append-only `identity_signals` ledger). Every other product is its own repo and a relying party that federates auth from it. noCluNetwork is one such product.

Two facts govern everything noCluNetwork does:

1. noCluNetwork federates auth and defers identity to noclulabs.com. It does not run its own auth, sessions, password storage, or OAuth2 provider, and it does not compute its own authenticity score. Identity, the dual authenticity score, and connection visibility all live in noCluID.

2. noCluNetwork is the bidirectional bridge between platforms and noCluID:
   - Write: it emits append-only authenticity signals up to the noCluID ledger (the intake contract). noCluNetwork is the ledger's first real writer.
   - Read: it reads scoped noCluID data back down (the read contract) so a bot can surface a user's identity on the platform they are standing in.

The other repos in the estate:

- noclulabs.com (exists): the auth, identity, and signal-ledger hub. The integration contracts (federation, intake, read) are governed by its AUTH-PLAYBOOK.md and IDENTITY-PLAYBOOK.md. Read those when a phase touches federation or the noCluID contract.
- noCluBot (separate repo, later): the multi-platform bot monorepo. A pure HTTP client of noCluNetwork over the service-token contract. It holds all platform adapters (Discord first) plus a shared adapter core, and it generates its typed client from noCluNetwork's OpenAPI spec.
- noCluCal (exists): the calendar product, a parallel relying party. Independent of noCluNetwork.

Repo-per-product is the rule. Each repo is born under the noclulabs org with its own four bible files.

## Bible files (canonical set)

Four bible files in the repo root are the source of truth for project state and the continuity mechanism between sessions, joined by two continuity files: RAMPUP.md (the stable architect entry point and pointer manifest) and HANDOFF.md (the volatile session baton, rewritten in full every session; see the HANDOFF.md Discipline section below). The executor ramp-up reading list is five files: Claude Code reads CLAUDE.md, README.md, ROADMAP.md, and HANDOFF.md in full at the start of every session, plus the `[Unreleased]` section of CHANGELOG.md (not its full released history; consult that on demand). Per-feature playbooks are added as subsystems grow and are read when the work touches that area.

Every executor prompt begins by reading the bibles and ends with a "Bible file updates (REQUIRED)" section naming the files and the edits. A PR that ships work without a CHANGELOG entry is drift the next session must fix. If a session finds drift or bloat, it surfaces it at the start and proposes an in-place fix (small) or a dedicated reclamation PR (large).

## HANDOFF.md Discipline

HANDOFF.md is the volatile session baton for architect continuity. Rules for every session:

1. The ramp-up reading list is five files: CLAUDE.md, README.md, ROADMAP.md, CHANGELOG.md, HANDOFF.md.
2. Before opening your PR, rewrite HANDOFF.md in full to reflect the state after your work. Overwrite the whole file. Never append.
3. Hard cap 4,000 characters. Verify with wc -c HANDOFF.md. If it does not fit, trim it. Roadmap detail belongs in ROADMAP.md, history belongs in CHANGELOG.md.
4. Rewriting HANDOFF.md is part of the definition of done for every session, the same as CHANGELOG.md updates. Skipping it is drift.
5. RAMPUP.md is stable. Do not modify RAMPUP.md unless your prompt explicitly instructs it.

## Tech stack

- Language: TypeScript, strict mode (no `any`, no `@ts-ignore`).
- Runtime: Node 22 LTS.
- API: Fastify, with `@fastify/cors`, `@fastify/rate-limit`, and `@fastify/swagger` plus swagger-ui (the OpenAPI spec is the contract noCluBot's client is generated from).
- Database: PostgreSQL 18 via Drizzle ORM (`pg` driver). Drizzle, not Prisma. The old code used Prisma; the estate standardizes on Drizzle for one ORM across products and the shared registry-as-canonical discipline (see the Database conventions below and noclulabs.com's DATABASE-PLAYBOOK.md).
- Cache, pub/sub, queues: Redis via ioredis, plus BullMQ (a separate ioredis instance for BullMQ to avoid the version-mismatch issue).
- Validation: Zod. Logging: Pino. Testing: Vitest.
- Package manager: pnpm. The old monorepo used npm; the estate standardizes on pnpm. noCluNetwork is a single application, not a workspace (a workspace is introduced only if a shared internal package is later warranted).
- Deployment: Docker on the shared noClu droplet, co-located with noclulabs.com in the noCluHub VPC, with its own database on the shared managed Postgres cluster and a dedicated Redis container. No public route today: bridge traffic is outbound over the private Docker network, and the summon endpoint is disabled and unexposed until noCluBot. A Caddy-fronted public surface on host port 3000 (the same TLS-terminating reverse proxy that also fronts noclulabs.com and noCluCal) is intended future state. CI and CD via GitHub Actions. See DEPLOYMENT.md for the full deployment and operational detail.

## Architectural invariants (load-bearing, do not drift)

1. Relying party, not issuer. Federate auth from noclulabs.com; never reimplement authentication, sessions, password storage, JWT issuance, or an OAuth2 provider. This is the whole reason for the rebuild. In the headless first scope, federation is service-to-service (see invariant 9). Browser-cookie suite SSO (AUTH_SECRET parity, the parent-domain cookie) applies only to a future noCluNetwork web portal, not the Core API.

2. Thin adapter, fat core. All business logic lives in noCluNetwork. Bots are pure HTTP clients over the service-token contract and run no business logic. The repo split (bots in noCluBot) makes this boundary physical: a bot cannot import core internals, only call the API.

3. Identity defers to noCluID. noCluNetwork keys platform accounts to noclulabs identities; it does not mint users and does not compute a second authenticity score. The dual authenticity score (true versus public) and the three connection visibility levels live in noCluID. noCluNetwork emits signals and reads scores; it never recomputes or stores an authenticity number.

4. The bidirectional bridge enforces privacy at the source. The noCluNetwork to noclulabs.com channel carries the intake write (append-only signals) and the scoped read (noCluID data for surfacing). The read API on noclulabs.com applies the dual-score and visibility model itself: the true score is owner-only and is only ever returned for the authenticated subject about themselves, never about another user, and bots deliver it privately (an ephemeral reply on Discord, a direct message where a platform has no ephemerals). The public score is the only shareable one. noCluNetwork and the bots never make a privacy decision and cannot leak; they render what the privacy-aware contract returns.

5. The verification linchpin. A platform account maps to a noclulabs identity only through a verified noCluID connection (a `connections` row on noclulabs.com with `provider = <platform>` and `provider_account_id = <platform user id>`). That single verification unlocks both directions: attributing the user's platform activity up as signals, and surfacing their noCluID back down. Discord as a noCluID provider is therefore a required noclulabs.com-side piece of this arc. Ghost accounts (auto-created from platform activity, not yet linked to a noclulabs identity) are the un-attributed state; a verification triggers claim-and-merge.

6. Registry-as-canonical. No DB metadata table and no foreign key onto registry ids; the TypeScript registry is the sole source of truth for the things it owns (signal types and their weights, qualifier definitions, and similar), with integrity enforced at the application layer. There is nothing to seed for those. This mirrors noclulabs.com.

7. Append-only ledgers and audit. The signal stream and any currency ledger ($IOC if kept) are append-only with a running snapshot, never mutated in place. Moderation and antispam outcomes are negative signals.

8. One Redis namespace from the first commit. The old code drifted across `pn:`, `cd:`, `xp_cd:`, `rl:`, `as:`, `user:`, and others. Unify on a single prefix (`ncn:`) for every key, everywhere.

9. Two service credentials. The service token (`X-Service-Token` plus `X-Service-Name`) authenticates bots to noCluNetwork. A separate trusted credential authenticates noCluNetwork to noclulabs.com for the intake and read contracts. Platform tokens (Discord and so on) live only in noCluBot, never in noCluNetwork.

## Conventions

### Code style
- TypeScript strict; no `any`, no `@ts-ignore`. Named exports; default export only where a framework requires it.
- Files kebab-case. A path alias maps to the source root.
- Each service owns its domain logic; routes are thin wrappers that parse, delegate, and render the response envelope.
- Response envelope: `{ success, data?, error?: { code, message }, pagination? }`, with a typed error class and a Fastify error handler. A serialization hook converts any BigInt to Number for JSON.

### Writing style
- No em dashes, anywhere, ever. Use commas, periods, or parentheses.
- Sentence case for headings and labels. No emoji. No exclamation marks. Plain, direct language. Minimal copy.

### Git conventions
- Conventional Commits: `type(scope): description`. Types: feat, fix, docs, refactor, test, chore, build, ci, perf, style, revert. Imperative mood, lowercase after the colon, no trailing period, under 72 chars.
- Branch naming: kebab-case, phase-named when part of a defined phase, otherwise descriptive with a prefix (`feat/`, `fix/`, `chore/`, `docs/`, `refactor/`).
- Squash merge to main.
- Versioning: calendar-semver `YYYY.MAJOR.MINOR.PATCH` (for example `2026.1.0.0`), applied as a git tag after merge, not committed as a file change. Tag when a meaningful unit ships, not every PR.

### Database (Drizzle)
- Schema in one file per table (kebab-case), with co-located inferred types. Migrations are append-only: never edit a migration applied to production, write a new one. Generate after editing schema files, apply locally before pushing.
- Drizzle does not auto-generate `CREATE EXTENSION` statements or triggers; any migration that needs them is hand-edited, the same way noclulabs.com does it.
- Production database access SSL workaround: every `DATABASE_URL` used by node-pg, drizzle-orm, or drizzle-kit must end with `&uselibpqcompat=true`, or the driver attempts `sslmode=verify-full` and fails against DigitalOcean's self-signed cert with `SELF_SIGNED_CERT_IN_CHAIN`. `psql` does not need the suffix. This is the same gotcha documented in noclulabs.com's DATABASE-PLAYBOOK.md.

### The OpenAPI contract
- noCluNetwork publishes an OpenAPI spec via `@fastify/swagger`. That spec is the contract noCluBot generates its typed client from, so it cannot silently drift. Keep route schemas accurate; the spec is a deliverable, not a side effect.

## Current state

The data-model foundation exists on top of the bootstrap. Five tables (participants, platform_accounts, communities, community_platforms, community_members) live in `src/lib/db/schema`, one file per table, established by the first migration (`drizzle/migrations/0000_*`), which is hand-edited to prepend the citext and pgcrypto extensions and the set_updated_at trigger and to add a per-table trigger. Ids default to the native Postgres 18 uuidv7(). participants is the spine: its `noclulabs_identity_id` is nullable-unique with no foreign key, and null means a ghost (auto-created from platform activity, not yet linked to a noclulabs identity). A ghost is verified, claimed, or merged through the claim endpoint (see below). The platform registry (`src/lib/registry/platforms.ts`) is the canonical, app-layer set of valid platforms (discord), projected into the route schemas. Two service-token resolve-or-create routes (`POST /api/v1/participants/resolve`, `POST /api/v1/communities/resolve`) are idempotent and race-safe, thin wrappers over services in `src/services`. Verification-driven claim-and-merge exists (`POST /api/v1/participants/claim`, `src/services/participants/claim.ts`): a verification assertion attaches a platform account to its noclulabs identity, claiming a ghost in place or merging it into the participant that already holds the identity. The survivor of a merge is always the identity-bearer. The merge relocates the loser's participant-owned rows onto the survivor, then the USER_HAS_DATA guard asserts the loser owns nothing before it is deleted; the relocation set and the guard share one centralized list (`src/services/participants/owned-relations.ts`), which holds platform_accounts, community_members, and moderation_actions, and which any future participant-owned table must be added to (a catalog assertion in the tests fails loud if a participant_id foreign-key table, or a second participant foreign-key column on a registered table, is left off). The claim's real verification source is now wired (phase 5, the verify-sync poller described below), which drives this exact function in-process from noclulabs.com's surface B; in tests both the claim and the poller are driven by synthetic assertions. The community membership lifecycle exists (phase 4a): `POST /api/v1/memberships/ensure` and `POST /api/v1/memberships/leave` (`src/services/memberships/lifecycle.ts`) compose the resolve-or-create paths and manage a community_members row with soft leave (the `active` and `left_at` columns, the 0001 migration) and rejoin (an inactive row reactivates in place, keeping created_at and permission_level). community_members is in the owned-relations list with a re-point-or-combine merge relocation: re-point where the survivor is not a member of the community, combine into the survivor's row where both are (higher permission_level, active if either, left_at accordingly), so the guard now covers memberships. permission_level defaults to 0 and stays uninterpreted (native moderation authorization is deferred, see moderation below). Network engagement leveling exists (phase 4b-i): a `network_xp` bigint column on participants (the 0002 migration) holds a participant's lifetime experience total across the whole internet of communities (account-wide and uncapped, never per-community in this slice). The leveling module (`src/lib/leveling/curve.ts`) is pure derive-at-read math: a no-maximum polynomial curve (`xpForLevel` is `floor((level / XP_SCALE) ^ EXPONENT)`, `levelForXp` its exact inverse with a float-drift guard at the thresholds; `XP_SCALE` 0.4879, `EXPONENT` 2.15) and a capped `trueScoreContribution` in [0, 1] that saturates at `CONTRIBUTION_CAP_LEVEL` 50. Level and contribution are derived from network_xp at read, never stored. `POST /api/v1/engagement` (`src/services/engagement/grant.ts`) is the call a bot makes on a trackable interaction: it composes ensure-membership, then grants a flat XP amount (default 20) gated by a per-community Redis cooldown (default 60s, a single ncn: key `engagement:cooldown:{communityId}:{participantId}` set with SET NX EX, the first real use of the Redis client), incrementing network_xp atomically. The merge sums `network_xp` into the survivor (network_xp is a participant-level scalar, not an owned child relation, so it is not in the owned-relations list). noCluNetwork stores engagement XP and computes a contribution value but never computes, stores, or names the True Score (that lives on noclulabs.com, computed from signals); the contribution is defined and tested here, and the emit client (below) now pushes it up as a capped authenticity signal. Moderation exists (phase 4c, the last of the engagement core, so phase 4 is complete): an append-only `moderation_actions` ledger (the 0003 migration, content-immutable, no updated_at and no trigger, no update or delete route) records one event per action with an actor, a target, the action, an optional reason, and an optional expires_at. The action registry (`src/lib/registry/moderation-actions.ts`) is the canonical app-layer set (warn, mute, unmute, kick, ban, unban). `POST /api/v1/moderation/actions` (`src/services/moderation/actions.ts`) resolve-or-creates the community, actor, and target, then in one transaction appends the log row and applies the membership effect (kick and ban soft-leave any membership via the shared phase 4a `deactivateMembershipRow`; a ban can target a non-member), returning the action and the target's derived sanction state. The sanction state (muted and banned, each with an optional until) is computed at read from the log and never stored (`src/services/moderation/sanction-state.ts`), exposed by two service-token reads: `GET /api/v1/moderation/state` and the paginated `GET /api/v1/moderation/history`. moderation_actions is the first owned relation with two participant foreign keys (actor and target), both re-pointed on a merge with no combine; the OwnedRelation contract now declares its participantColumns and the catalog assertion cross-checks them against the catalog. Native moderation authorization is deferred by design: the route is service-token gated, the trusted bot enforces platform-side moderator permission, and the core records the reported actor without interpreting permission_level. The verify-sync poller exists (phase 5, the bridge verify capability, the first realized consumer of the noCluID bridge on our side): a background poller (`src/services/verify-sync/`) reads verified Discord connections from noclulabs.com's surface B (the locked v3 bridge contract, behind the `src/lib/noclulabs/` integration boundary: an authed, timeout-bounded client and a Zod-validated `VerifiedConnectionsClient` port that the poller depends on, so the test path injects a fake and touches no network) and drives the existing claim service in-process for each, mapping `(provider, providerAccountId, noclulabsIdentityId)` one to one onto the claim tuple. It does not reimplement claim logic and never calls our own HTTP endpoint. A fast-path incremental cycle advances a durable Postgres watermark (the `sync_watermarks` table, the 0004 migration, keyed by stream so other platforms slot in later) page-atomically: the cursor persists only after a whole page is processed, so a transient error leaves it unadvanced and the next cycle safely re-fetches the same page (failure-no-advance, free because the claim is idempotent). A periodic full re-scan sweeps from the beginning to close the commit-order gap surface B's cursor cannot (replay-safe but not gap-free under out-of-order commits), never touching the fast-path cursor, then stamps full_rescan_at. The scheduler plugin (`src/plugins/verify-sync.ts`) is config-gated off by default (`VERIFY_SYNC_ENABLED`, plus `VERIFY_SYNC_INTERVAL_MS`, `VERIFY_SYNC_RESCAN_INTERVAL_MS`, `VERIFY_SYNC_PAGE_SIZE`, `NOCLULABS_HTTP_TIMEOUT_MS`): it self-gates to start no timers and touch neither database nor network when disabled, so the feature ships inert and the test environment never starts it; config load fails fast if the poller is enabled without `NOCLULABS_BASE_URL` and `NOCLULABS_SERVICE_TOKEN`. The watermark is Postgres-durable and the `ncn:` Redis namespace is untouched; sync_watermarks has no participant foreign key, so it is not an owned relation and the catalog assertion is unaffected. The emit capability of the bridge also exists (phase 5, the bridge emit, surface A): when a claimed participant's network level changes, noCluNetwork pushes their capped leveling contribution up to noclulabs.com's signal intake (`POST /api/identity/signals`, bearer-gated, the locked v3 contract). The emit client (`src/lib/noclulabs/signals.ts`) mirrors the poller and reuses the same authed base (`client.ts`, refactored to share a GET and a POST core, token never logged): a `SignalsClient` port whose `emit` POSTs `{ subjectIdentityId, signalType: "network.level", value, observedAt }` (value is `trueScoreContribution` = `min(level, 50) / 50`; the server sets source itself), Zod-validates the response, and returns a typed outcome distinguishing success (`written`), the two surface-A 422s (`unknown_subject`, `invalid_request`), and the base's thrown transport, auth, and server errors. The emit orchestration (`src/services/emit-sync/emit.ts`) is best-effort and post-commit: it gates on `EMIT_SYNC_ENABLED`, loads the participant, skips a ghost (no subject), a missing participant (a merged ghost), or a confirmed stale link, derives the value from the existing `trueScoreContribution` (never reimplemented), emits, and swallows every error (logged) so it never throws into a caller. On `unknown_subject` (the noclulabs.com subject was deleted) it sets a permanent stale-link marker, the nullable `identity_emit_disabled_at` column on participants (the 0005 migration, additive, no backfill), so that subject never emits again; `invalid_request` is a request bug and never sets the marker (the stale path keys off `unknown_subject` only). Three events emit, always after the relevant transaction commits: an engagement level-up (only when the integer level actually crosses, not on every XP gain), a claim, and a merge (the survivor's summed contribution). Claim and merge route through one shared claim-and-emit wrapper (`src/services/emit-sync/claim-and-emit.ts`) used by both the claim route and the verify-sync poller, so there is a single emit trigger; the wrapper preserves the poller's page-atomic, failure-no-advance behavior exactly (only a claim error propagates, a failing emit is swallowed in the orchestration and never alters the poller's advance). The emit ships inert (`EMIT_SYNC_ENABLED` false by default, config fails fast if enabled without `NOCLULABS_BASE_URL` and `NOCLULABS_SERVICE_TOKEN`) and is tested offline with an injected fake `SignalsClient`. The emit reconcile backstop also exists (phase 5, the emit follow-up): a scheduled, stateless full pass (`runReconcileCycle` in `src/services/emit-sync/reconcile.ts`, scheduled by `src/plugins/emit-reconcile.ts`) keyset-paginates through claimed, non-stale participants (`noclulabs_identity_id` not null and `identity_emit_disabled_at` null, ordered by id, in bounded batches so it never loads all at once) and runs the same `emitParticipantContribution` orchestration on each, so any emit the on-event path failed to land (noclulabs.com unreachable, or a crash between commit and the best-effort emit) eventually lands. It reimplements nothing and adds no schema (it reuses the 0005 stale-link marker). It is stateless (no watermark, a fresh full pass each cycle) and cheap because the server conditionally appends (a participant already current writes nothing, `written` false); it is best-effort per participant (the orchestration swallows its own emit errors) and stops a cycle only on a query or database error, retried next interval. A participant that returns `unknown_subject` mid-pass is marked by the orchestration, already passed in the forward scan, and excluded by the next cycle's query, so there is no loop. The scheduler (`src/plugins/emit-reconcile.ts`) is config-gated off (`EMIT_RECONCILE_ENABLED`, plus `EMIT_RECONCILE_INTERVAL_MS` default six hours and `EMIT_RECONCILE_BATCH_SIZE` default 200), starts a single overlap-guarded, unref'd timer only when both `EMIT_SYNC_ENABLED` and `EMIT_RECONCILE_ENABLED` are true (the reconcile re-emits through the same orchestration, so it is meaningless without the on-event emit), and self-gates to start nothing otherwise; config load fails fast if the reconcile is enabled without the base URL and token. The `ncn:` Redis namespace is untouched. The summon endpoint exists (phase 5, the bridge read-down capability, the last of the three noCluNetwork-side bridge sessions): `POST /api/v1/summon` (`src/routes/summon.ts`, `src/services/summon/summon.ts`), service-token gated inbound, resolves an invoking platform user to their claimed participant READ-ONLY (a lookup by (platform, platformUserId) and its participant, never the resolve-or-create path, never creating or writing participant state) and reads that subject's noCluID score from noclulabs.com's surface C (`GET /api/identity/score`, bearer-gated, the locked v3 contract, live on their side). The score client (`src/lib/noclulabs/score.ts`) mirrors the emit and reuses the same authed base via a new `noclulabsGetReadable` (a GET that returns the status and body for a 2xx or a readable 422, alongside the unchanged `noclulabsGet`): a `ScoreClient` port whose `fetchScore({ subject, actingForSubject })` sends `actingForSubject` as the exact lowercase literal (typed as the union `"true" | "false"`, never `String(boolean)` or a template, so a non-canonical value like `True` that surface C rejects with 422 can never be sent), validates the response strictly on the totals and leniently on the breakdown passthrough (an object surfaced onward), and returns a typed outcome (`ok`, the two 422s `unknown_subject`/`invalid_request`, and the base's thrown transport, auth, and server errors). The summon service depends on the port (so the suite runs offline) and, unlike the emit, does not swallow failures: it maps every case to a defined outcome. It always sends `actingForSubject` `"true"` (the invoking user is the subject themselves, so they get their owner-only true score; both scores come from the one call). It unifies never-seen and unclaimed-ghost into one `not_linked` outcome (logging which sub-case), maps `unknown_subject` to a distinct `subject_gone` (read-only, never setting the stale marker; the emit path owns it and self-heals), `invalid_request` to an internal error (our request bug), and any thrown transport/auth/server error to an upstream error. The route gates inbound with the existing service-auth plugin (a 401 always, independent of the flag), then returns 503 `summon_disabled` when `SUMMON_ENABLED` is off (no resolution, no surface C call), validates the body against the platform registry (a malformed body is a 422, this endpoint's surface-C-aligned semantics, via `attachValidation` keeping the OpenAPI body schema accurate while the Zod gate enforces it and `parseOrThrow` gained an optional status-code arg defaulting to 400; note this 422 differs from the 400 every other route returns), and maps the outcome to the standard envelope: `ok`/`not_linked`/`subject_gone` are 200 with a `data.outcome` discriminator (and on `ok` the subject and both scores), an internal error is a 500, an upstream error a 502. The inbound service token (`SERVICE_TOKEN`, a bot calling us) stays distinct from the outbound credential (`NOCLULABS_SERVICE_TOKEN`, us calling noclulabs.com). It ships inert (`SUMMON_ENABLED` false by default, config fails fast if enabled without `NOCLULABS_BASE_URL` and `NOCLULABS_SERVICE_TOKEN`), is tested offline (an injected fake `ScoreClient` for the endpoint, a spied fetch for the on-the-wire acting-for-subject literal), and touches no schema and no `ncn:` Redis namespace. With verify, emit, reconcile, and read-down all built, the noCluNetwork side of the bridge is structurally complete; presentation (the Discord command and its private ephemeral/DM delivery) is noCluBot's job. As of 2026-06-30 the phase 5 bridge is deployed and live in production and verified in both directions: the operational wiring (the shared token on both sides, the private base URL) is in place, and the production flags are verify, emit, and reconcile enabled with summon disabled pending noCluBot. In production noCluNetwork runs co-located on the droplet in the noCluHub VPC, with its own database on the shared managed Postgres cluster and a dedicated Redis container, and reaches noclulabs.com over the private Docker network; there is no public route today (summon exists but stays disabled and unexposed until noCluBot). The remaining work is the noCluBot consumer (phase 6). DEPLOYMENT.md is the authoritative deployment and operations record. The dev Postgres host port is 5439 (5433 belongs to the noclulabs.com dev Postgres). ROADMAP.md is authoritative for phase status; CHANGELOG.md holds the history.

## Gotchas and do-not-touch

- Do not add native auth, password storage, a JWT issuer, or an OAuth2 provider. Identity and auth defer to noclulabs.com (invariants 1 and 3).
- Do not put business logic in a bot or let a bot reach any datastore directly. Bots call the API (invariant 2).
- Do not compute or store an authenticity score in noCluNetwork. Emit signals, read scores (invariant 3).
- Do not return a true score for anyone other than the authenticated subject, and do not let a bot surface owner-only data in a public context (invariant 4).
- Do not split the Redis namespace; one prefix (invariant 8).
- Do not drop the `&uselibpqcompat=true` suffix from a production `DATABASE_URL`.
- Do not add a participant-owned table (or a second participant foreign-key column on one) without adding it to the centralized participant-owned-relations list (`src/services/participants/owned-relations.ts`, which holds platform_accounts, community_members, and moderation_actions). The merge relocation and the USER_HAS_DATA guard both iterate that list; a table or column left off it can be silently dropped when a ghost is merged (the participant_id foreign keys cascade on delete). Each entry declares its participantColumns (moderation_actions has two, actor and target, both re-pointed on a merge). The owned-relations catalog assertion (`test/db/owned-relations-catalog.test.ts`) fails loud if a participant foreign-key table or column is ever left off, but treat that as the backstop, not the reason to skip the discipline.
- Do not add native moderation authorization: no interpreting permission_level, no role assignment or sync, no gating who may moderate. The moderation route is service-token gated and trusts the bot to enforce platform-side moderator permission; the core records the reported actor. This is deferred by design until a non-bot surface exists, not an omission.
- Do not have the verify-sync poller reimplement claim logic or call our own HTTP endpoint; it is a relying-party consumer that drives the existing claim service in-process. Do not change surface B (it is locked and live on the noclulabs.com side) or the claim contract from the poller. Do not enable the poller by default (it ships off, `VERIFY_SYNC_ENABLED` false), and do not move its watermark off Postgres into the `ncn:` Redis namespace; the watermark must survive restarts, and the namespace stays reserved for the engagement cooldown (invariant 8).
- Do not let the emit reimplement the leveling curve or the contribution math (call `trueScoreContribution`), and do not change surface A (`POST /api/identity/signals`) or the claim contract from the emit; only call them. Do not run an emit inside a transaction or let it throw into a triggering operation; emits are post-commit and best-effort (the orchestration swallows every error and only ever sets the `identity_emit_disabled_at` marker on `unknown_subject`, never on `invalid_request`). Do not alter the poller's page-atomic, failure-no-advance behavior through the shared claim-and-emit wrapper. Do not enable the emit by default (it ships off, `EMIT_SYNC_ENABLED` false), and do not use the `ncn:` Redis namespace for it.
- Do not reimplement the emit orchestration, the contribution math, or the emit client in the reconcile (`src/services/emit-sync/reconcile.ts`); it only calls the existing `emitParticipantContribution` per participant. Do not make the reconcile stateful (no watermark, no new schema; it is a stateless full pass reusing the 0005 marker), do not let a cycle throw into anything that can crash the process (the scheduler guards every cycle with try/catch and an overlap flag), and do not alter the on-event emit path or the verify-sync poller through it. Do not enable the reconcile by default (it ships off, `EMIT_RECONCILE_ENABLED` false, and the scheduler starts only when `EMIT_SYNC_ENABLED` is also true), and do not use the `ncn:` Redis namespace for it.
- Do not make the summon create a participant or write any participant state; it is read-only (a lookup, never the resolve-or-create path), and even `subject_gone` writes nothing (the emit path owns the `identity_emit_disabled_at` marker and self-heals). Do not change surface C (`GET /api/identity/score`, locked and live on the noclulabs.com side) or reimplement auth; reuse the inbound service-auth plugin and the outbound noclulabs base, and keep the inbound `SERVICE_TOKEN` and outbound `NOCLULABS_SERVICE_TOKEN` distinct. Do not send a coerced or template-serialized `actingForSubject`; it is the exact lowercase literal (the `"true" | "false"` union), always `"true"` for the summon (the invoking user is the subject). Do not enable the summon by default (it ships off, `SUMMON_ENABLED` false), do not build the Discord command or any presentation layer here (that is noCluBot), and do not use the `ncn:` Redis namespace for it.

## The git-sync-first rule (from PR 2 on)

Every executor prompt after the bootstrap opens with the git sync sequence (status, fetch, checkout main, pull, status) before any other Mac command. The bootstrap PR is the one exception, because the repo does not exist until it runs.
