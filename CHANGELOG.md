# Changelog

All notable changes to noCluNetwork are recorded here, in Keep a Changelog format, grouped under Added, Changed, Fixed, and Removed. The project uses calendar-semver `YYYY.MAJOR.MINOR.PATCH`, applied as a git tag when a meaningful unit ships. Unreleased work accumulates under the [Unreleased] heading until it is tagged.

## [Unreleased]

### Added

- DEPLOYMENT.md, the production deployment runbook and the authoritative record of how noCluNetwork is deployed and brought live (the topology, the managed-Postgres database and its privilege prerequisites, the migration mechanics, the four bridge flags, the staged go-live runbook, and the operational gotchas). It records the milestone: the phase 5 bridge was deployed to production and verified live in both directions, with the inbound verify poller and the outbound emit both proven live (stages 0 through 2 of the runbook) and the production flags verify, emit, and reconcile enabled with summon disabled pending noCluBot. An operational milestone, not a code change.
- The summon endpoint (phase 5, the bridge read-down capability), the last of the three noCluNetwork-side bridge sessions, which completes the noCluNetwork half of the bridge: verify, emit, and read-down are now all built. `POST /api/v1/summon` (service-token gated inbound) resolves an invoking platform user to their claimed participant, read-only, and reads that subject's noCluID score from noclulabs.com surface C, returning the true score and the public score for a bot to present privately. The Discord bot command and its private (ephemeral or DM) presentation live in noCluBot (a future repo); this endpoint is the unit noCluBot calls. It ships inert behind `SUMMON_ENABLED` (off by default), returning 503 when off and never calling surface C, so merging it changes nothing in production.
- The score-read client (`src/lib/noclulabs/score.ts`) and its `ScoreClient` port, the inbound read half of the bridge. `fetchScore({ subject, actingForSubject })` calls surface C (`GET /api/identity/score`, bearer-gated) and returns a typed outcome: success (`ok`, carrying the subject, the public score, and the true score when present), a stale subject (422 `unknown_subject`), and an invalid request (422 `invalid_request`), with the base client's thrown transport, auth (401), and server (500) errors left as `NoclulabsClientError`. `actingForSubject` is typed as the literal union `"true" | "false"` and sent to the wire verbatim, never built from `String(boolean)` or a template, so a non-canonical value (for example `True`, which surface C rejects with 422) can never be sent. The response is validated strictly on the totals (`publicScore.total`, `trueScore.total` as numbers) and leniently on the breakdown (a present object surfaced onward without depending on its internal shape, so a future noclulabs bucket change does not break the parse). The summon service depends on the port, so the suite runs fully offline against an injected fake.
- The readable-status GET helper on the noclulabs base client (`src/lib/noclulabs/client.ts`), `noclulabsGetReadable`, which returns the status and parsed body for a 2xx or for any readable non-2xx the caller passes (surface C's 422), and throws a typed `NoclulabsClientError` for 401, 500, any other non-2xx, or a transport, timeout, or non-JSON failure. It mirrors `noclulabsPost` and reuses the same authed core, so the score client does not duplicate the auth, timeout, or error mapping. The existing `noclulabsGet` (body-only, throws on any non-2xx) is unchanged, so the verify-sync poller is unaffected.
- The summon service (`src/services/summon/summon.ts`), which given a `platform` and `platformUserId` returns a typed outcome. It resolves the platform account and its participant read-only (never creating a participant), and unifies the never-seen and unclaimed-ghost cases into one `not_linked` outcome (logging which sub-case occurred). For a claimed participant it calls the score client with the subject set to its `noclulabs_identity_id` and `actingForSubject` `"true"` (the invoking user is the subject), then maps the result: success to `ok` (both scores from one call), `unknown_subject` to `subject_gone` (the identity was deleted on noclulabs.com, distinct from `not_linked`), `invalid_request` to an internal error (our request bug), and any thrown transport, auth, or server error to an upstream error. Unlike the emit it does not swallow failures; it maps every case to a defined outcome. It is read-only by design and writes no participant state, including on `subject_gone` (the emit path remains the authority for the stale-link marker and self-heals). It depends on a `ScoreClient` and an enabled flag through a runtime seam mirroring the emit, so the suite drives it offline.
- The summon route (`src/routes/summon.ts`), `POST /api/v1/summon`, gated by the existing inbound service-auth plugin (a missing or bad service token is the plugin's 401, always required and independent of the flag). After the auth gate it returns 503 `summon_disabled` when `SUMMON_ENABLED` is off (doing no resolution and never calling surface C), validates the body against the platform registry (a malformed body is a 422, this endpoint's surface-C-aligned unprocessable-entity semantics, via `attachValidation` so the OpenAPI body schema stays accurate while the Zod gate enforces it), and maps the service outcome to the response envelope: `ok`, `not_linked`, and `subject_gone` are 200 with a `data.outcome` discriminator (and on `ok` the subject and both scores); an internal error is a 500 (`internal`) and an upstream error is a 502 (`upstream_error`), so a failure surfaces loudly rather than masquerading as a normal result. The registry-derived platform enum keeps the OpenAPI spec from drifting.
- The summon config addition: `SUMMON_ENABLED` (boolean, default false, the endpoint gate), reusing `NOCLULABS_BASE_URL`, `NOCLULABS_SERVICE_TOKEN`, and `NOCLULABS_HTTP_TIMEOUT_MS` from the poller and emit, with a refine that fails config load fast if the summon is enabled while the base URL or service token is missing (the same posture as the poller, emit, and reconcile refines). The inbound endpoint gate and the inbound service token (`SERVICE_TOKEN`, a bot calling us) stay distinct from the outbound credential (`NOCLULABS_SERVICE_TOKEN`, us calling noclulabs.com). `parseOrThrow` gained an optional status-code parameter (defaulting to 400, so every existing caller is unchanged) so the summon can return a 422.
- Summon tests: a DB-backed suite (`test/db/summon.test.ts`) that runs fully offline against the test Postgres with an injected fake `ScoreClient`, covering the claimed path returning both scores from a single call with the subject equal to the participant's identity, the `actingForSubject` literal `"true"` at the port, the unified `not_linked` for a never-seen user (no participant created) and an unclaimed ghost (no surface C call), the distinct `subject_gone` with no participant state written, the `invalid_request` to 500 and the thrown-upstream (401, 500, unexpected status, network) to 502 mapping, the inbound service-token auth (missing and wrong), the disabled 503 with no surface C call, and body validation (missing field and unregistered platform) to 422; a score-client suite (`test/score-client.test.ts`) that drives the real client against a spied global fetch and asserts the on-the-wire acting-for-subject literal is exactly `true` (never `True`/`TRUE`), the bearer in the header and never the URL, and the 200, 422, 401, network, and malformed-body parses; and a config-and-gating suite (`test/summon-config.test.ts`) covering the default off, the enable path, the fail-fast refine, and the boolean validation. There is no migration in this session.
- The emit reconcile backstop (phase 5, the bridge emit follow-up), a scheduled, stateless full pass that re-emits every claimed participant's current leveling contribution to noclulabs.com surface A through the existing emit orchestration. It closes the two gaps the on-event emit cannot cover: an emit lost while noclulabs.com was unreachable, and a crash between a committed transaction and its best-effort emit. It reimplements nothing (no orchestration, no contribution math, no emit client) and adds no schema, reusing the stale-link marker from migration 0005. Like the on-event emit it ships inert behind its own flag.
- The reconcile job (`src/services/emit-sync/reconcile.ts`), a function `runReconcileCycle` that keyset-paginates through claimed, non-stale participants (`noclulabs_identity_id` not null and `identity_emit_disabled_at` null, ordered by id ascending, `id > lastSeenId` with a bounded limit) and runs `emitParticipantContribution` on each, so the pass never loads all participants at once. The orchestration stamps `observedAt` from its own clock (the current time), which is correct for a now-observation, and derives the value from the existing `trueScoreContribution`. A full re-emit is preferred over tracking staleness because the server conditionally appends: a participant already current writes nothing and returns `written` false, so a full pass is cheap on the ledger and self-correcting, with no per-participant bookkeeping. It is best-effort per participant (the orchestration swallows its own emit errors, so one participant's failure does not stop the pass) and stateless (each cycle is a fresh full pass from the beginning, no watermark). A selection-query or database error propagates out and stops the cycle; the next interval retries from the beginning. A participant that returns `unknown_subject` mid-pass gets its stale marker set by the orchestration, has already been passed in the forward scan, and is excluded by the next cycle's query, so there is no loop or double processing. The default selection query and the per-participant emit are injectable seams (defaulting to the real query and the real orchestration) so the suite proves the query-level skips and a selection-query error offline.
- The emit reconcile scheduler plugin (`src/plugins/emit-reconcile.ts`), registered in the app factory but self-gating: it starts a timer only when both `EMIT_SYNC_ENABLED` and `EMIT_RECONCILE_ENABLED` are true (the reconcile re-emits through the same orchestration and signals client, so it is meaningless without the on-event emit), otherwise it registers and does nothing (no timer, no database, no network), which keeps a merge safe and keeps the test environment from ever starting it. When enabled, on the app ready hook it starts a single recurring interval (`EMIT_RECONCILE_INTERVAL_MS`) running `runReconcileCycle`, with an in-flight flag so a slow pass never overlaps, a try/catch so a failed cycle is logged and never crashes the process or stops future cycles, and an unref'd timer cleared on close. Single instance for v1; multi-instance coordination is deferred (concurrent reconciles are safe because the emit is idempotent, just wasteful). The `ncn:` Redis namespace is untouched.
- The emit reconcile config additions: `EMIT_RECONCILE_ENABLED` (boolean, default false, the gate), `EMIT_RECONCILE_INTERVAL_MS` (default 21600000, six hours), and `EMIT_RECONCILE_BATCH_SIZE` (default 200, validated 1 to 1000), with a refine that fails config load fast if the reconcile is enabled while `NOCLULABS_BASE_URL` or `NOCLULABS_SERVICE_TOKEN` is missing (the same posture as the emit and poller refines). While the flag is false nothing reconciles and nothing touches noclulabs.com.
- Emit reconcile tests: a DB-backed suite (`test/db/emit-reconcile.test.ts`) that runs fully offline against the test Postgres with an injected fake `SignalsClient`, covering re-emitting an unlanded contribution with the correct subject and value, the dedup-cheap no-op (`written` false), pagination across batches with the keyset advancing, query-level skips of ghosts and stale links (proved with an injected emit spy), best-effort isolation (one participant's emit failure does not stop the pass), the stale marker set on `unknown_subject` and the subject excluded from the next pass with no loop, the stateless full re-scan each cycle, a selection-query error stopping the cycle then a clean retry, and the emit-disabled no-op; plus a config-and-gating suite (`test/emit-reconcile-config.test.ts`) covering the defaults, the enable path, the fail-fast refine, the boolean and batch-size validation, and the scheduler registering no lifecycle hooks unless both flags are on.
- The emit client (phase 5, the bridge emit capability), the outbound half of the noCluID bridge and the signal ledger's first real writer from this side. When a claimed participant's network level changes, noCluNetwork pushes their capped leveling contribution up to noclulabs.com's signal intake (surface A of the locked v3 bridge contract, `POST /api/identity/signals`, bearer-gated). The emit is best-effort and always fires after the relevant transaction commits, never inside it, so a failed emit never endangers the XP gain, the claim, the merge, or a poller cycle. With this slice the bridge emits real signals from the noCluNetwork side. The on-event emit is backed by the reconcile backstop below, which re-emits on a schedule to recover any emit the on-event path failed to land.
- The emit client module (`src/lib/noclulabs/signals.ts`) and its `SignalsClient` port. `emit` POSTs `{ subjectIdentityId, signalType: "network.level", value, observedAt }` to surface A (the server sets `source` itself, never sent), Zod-validates the response, and returns a typed outcome that distinguishes success (200 `{ ok, written }`, where `written` false is an unchanged-value no-op), a stale subject (422 `unknown_subject`), and an invalid request (422 `invalid_request`), with the base client's thrown transport, auth (401), and server (500) errors left as `NoclulabsClientError`. The two 422s are returned (not thrown) because the caller acts on each differently, and the stale path keys off `unknown_subject` specifically so a request bug never disables a subject. The orchestration depends on the port, so the suite runs fully offline against an injected fake.
- The emit orchestration (`src/services/emit-sync/emit.ts`): a single best-effort, post-commit function that gates on `EMIT_SYNC_ENABLED`, loads the participant, skips a ghost (no noclulabs subject), a missing participant (a merged ghost), or a confirmed stale link, derives the value from the existing `trueScoreContribution` (`min(level, 50) / 50`, never reimplemented), emits, and on `unknown_subject` sets the permanent stale-link marker. Every error is swallowed (logged) so the emit never throws into its caller. An injectable runtime seam supplies the real signals client and the config flag in production and a fake client (enabled, with a silent logger) in tests, so the orchestration itself is exercised offline.
- The three post-commit emit triggers, each firing only after the relevant transaction commits: an engagement level-up (`src/services/engagement/grant.ts`, only when the integer level actually crosses, not on every XP gain, honoring moves-at-level-up), a claim, and a merge (the survivor's summed contribution). The claim and merge triggers route through one shared claim-and-emit wrapper (`src/services/emit-sync/claim-and-emit.ts`) used by both the claim route and the verify-sync poller, so there is a single emit trigger point. The wrapper performs the claim and then, post-commit, runs the emit orchestration on the resulting participant; only a claim error can propagate from it, so the poller's page-atomic, failure-no-advance behavior is preserved exactly and a failing emit never alters the poller's advance.
- The stale-link marker and the sixth migration (`drizzle/migrations/0005_old_shotgun.sql`, a plain append-only ALTER adding one nullable column with a null default, so a safe additive migration with no backfill; 0000 through 0004 untouched, and the existing set_updated_at trigger already maintains updated_at). The `identity_emit_disabled_at` column on participants records that a prior emit returned `unknown_subject` (the noclulabs.com subject was deleted, made permanent by the FK cascade on their side); a non-null value permanently stops emitting for that subject. It is a participant-level scalar (not an owned child relation), so it does not join the owned-relations list and the catalog assertion is unaffected. It is distinct from unlinking a platform connection (out of scope), which leaves the user intact.
- The emit config addition: `EMIT_SYNC_ENABLED` (boolean, default false, the gate), reusing `NOCLULABS_BASE_URL`, `NOCLULABS_SERVICE_TOKEN`, and `NOCLULABS_HTTP_TIMEOUT_MS` from the poller, with a refine that fails config load fast if the emit is enabled while the base URL or service token is missing (the same posture as the poller refine). While the flag is false no triggering event emits and nothing touches noclulabs.com.
- Emit tests: a DB-backed suite (`test/db/emit-sync.test.ts`) that runs fully offline against the test Postgres with an injected fake `SignalsClient`, covering the level-crossing gate (a crossing emits once with the new level's value, a non-crossing emits nothing), value correctness through the cap (level 50 and above emit 1.0), claim and merge emits with the right subject and value, the single shared wrapper across both the claim route and the poller paths, best-effort isolation (a failing emit fails no engagement, claim, or merge, and does not stop or alter a poller cycle that still advances its watermark page-atomically), the stale-link marker set on `unknown_subject` and not on `invalid_request`, ghost and disabled skips, and config gating (with the emit off, no event emits and a fetch spy confirms nothing reaches the network); plus a config suite (`test/emit-sync-config.test.ts`) covering the default off, the enable path, the fail-fast refine, and the boolean validation.
- The verify-sync poller (phase 5, the bridge verify capability), the first realized consumer of the noCluID bridge on the noCluNetwork side. A background poller reads verified Discord connections from noclulabs.com (surface B of the locked v3 bridge contract) and drives the existing phase 3 participant claim service in-process for each one, linking ghost participants to their noCluID. It ships inert (feature-flagged off by default), so merging it changes nothing in production until an operator enables it.
- The `sync_watermarks` table and the fifth migration (`drizzle/migrations/0004_cute_winter_soldier.sql`, a plain CREATE TABLE hand-edited to append the set_updated_at trigger reusing the 0000 trigger function; 0000 through 0003 untouched). It is the durable, restart-safe cursor state for the poller, keyed by `stream` (text primary key, for example `noclulabs:verified:discord`), with a nullable `cursor` (the last consumed connection id, null means from the beginning), a nullable `full_rescan_at` (when the last full re-scan completed, used to gate the re-scan cadence across restarts), and created_at and updated_at. The watermark is Postgres-durable, not Redis: the single `ncn:` namespace stays reserved for the engagement cooldown and is untouched. The table has no foreign key to participants, so it is intentionally absent from the participant-owned-relations list and the catalog assertion is unaffected.
- The noclulabs.com integration boundary (`src/lib/noclulabs/`), the first real use of `NOCLULABS_BASE_URL` and `NOCLULABS_SERVICE_TOKEN`. A base client (`client.ts`) provides an authed GET that attaches the bearer (never logged), enforces `NOCLULABS_HTTP_TIMEOUT_MS` via an AbortController, and maps the response into the parsed body or a typed `NoclulabsClientError` whose `kind` distinguishes 401 (unauthorized), 500 (server_misconfigured), any other non-2xx (unexpected_status), and a transport, timeout, or non-JSON failure (network). A verified-connections module (`verified-connections.ts`) exposes the `VerifiedConnectionsClient` port the poller depends on (so the test path injects a fake and touches no network), whose `fetch` method builds the surface-B query and Zod-validates the response body before returning a typed `{ connections, nextCursor }` (the wire shape is not trusted; the identity id is validated as a uuid since the poller bypasses the route's own body validation).
- The poller itself (`src/services/verify-sync/`), a factory over injected dependencies (the client port, the claim service as a function, a watermark store, a logger) so it is fully testable offline. A shared page-processing routine drives the idempotent claim for each surface-B row in order, mapping the tuple `(provider, providerAccountId, noclulabsIdentityId)` one to one onto the claim tuple `(platform, platformUserId, noclulabsIdentityId)`. `runIncrementalCycle` reads the watermark, drains pages from it, and advances the watermark page-atomically: the cursor is persisted only after a whole page is processed without a thrown error, so a transient error leaves it unadvanced and the next cycle re-fetches and re-processes the same page safely (failure-no-advance, free because the claim is idempotent). `runFullRescan` sweeps from the beginning with a local cursor, never reading or writing the fast-path cursor, then stamps `full_rescan_at`; it is the gap-closure mechanism for surface B, whose cursor is replay-safe but not gap-free under out-of-order commits. A normal claim outcome (claimed, already_linked, merged) is counted; the defined ACCOUNT_ALREADY_VERIFIED conflict and an unregistered-platform row are logged and counted as anomalies and do not stall the stream; any other error rethrows and stops the cycle without advancing. The watermark store (`watermark.ts`) reads a stream's row (creating it on first use) and upserts the cursor and the full-rescan timestamp.
- The verify-sync scheduler plugin (`src/plugins/verify-sync.ts`), registered in the app factory but self-gating: with `VERIFY_SYNC_ENABLED` false it starts no timers and touches neither the database nor the network (so the test environment never starts it). When enabled, on the app ready hook it constructs the real client and poller and starts a fast-path interval (`VERIFY_SYNC_INTERVAL_MS`) and a full re-scan (`VERIFY_SYNC_RESCAN_INTERVAL_MS`, its first run gated on `full_rescan_at` so a restart does not reset the re-scan clock). In-flight flags prevent overlapping runs of each kind, every tick is wrapped so a failed cycle is logged and never crashes the process or stops future cycles, and the timers are unref'd and cleared on close. Single instance for v1; a Redis advisory lock for multi-instance safety is deferred (concurrent pollers are safe because the claim is idempotent, just wasteful).
- The verify-sync config additions: `VERIFY_SYNC_ENABLED` (boolean, default false, the primary gate), `VERIFY_SYNC_INTERVAL_MS` (default 60000), `VERIFY_SYNC_RESCAN_INTERVAL_MS` (default 3600000), `VERIFY_SYNC_PAGE_SIZE` (default 200, validated 1 to 500), and `NOCLULABS_HTTP_TIMEOUT_MS` (default 10000), with a refine that fails config load fast if the poller is enabled while `NOCLULABS_BASE_URL` or `NOCLULABS_SERVICE_TOKEN` is missing. While the flag is false the URL and token stay optional and the poller stays inert.
- Verify-sync tests: a DB-backed suite (`test/db/verify-sync.test.ts`) that runs fully offline against the test Postgres with an injected fake surface-B client returning synthetic tuples (surface B is empty in production until real Discord verifications exist), covering a single cycle, the pagination drain with per-page cursor advance, idempotent re-run with no duplicates, the gap-closure re-scan sweeping a deliberately skipped row without disturbing the fast-path cursor, watermark persistence and read-back, failure-no-advance then a clean retry, tuple mapping with the account verified, the conflict anomaly counted-and-continued, and the unregistered-platform anomaly skipped; plus a config-and-gating suite (`test/verify-sync-config.test.ts`) covering the defaults, the enable path, the fail-fast refine, the boolean and page-size validation, and the scheduler registering no lifecycle hooks and touching no network when the flag is off.
- Moderation actions and sanction state (phase 4c), the moderation slice and the last of the community engagement core. With it, phase 4 (the community engagement core) is complete. An append-only moderation log drives a sanction state computed at read, with no native authorization: the trusted bot enforces platform-side moderator permission, and the core records the reported actor.
- The `moderation_actions` table and the fourth migration (`drizzle/migrations/0003_jittery_inhumans.sql`, a plain append-only CREATE TABLE; 0000 through 0002 untouched). It is an append-only ledger of content-immutable rows, so it deliberately carries no updated_at and no set_updated_at trigger, and there is no update or delete route. Columns: community_id, actor_participant_id, target_participant_id (all not-null foreign keys, ON DELETE CASCADE), action (free-form text validated against the action registry at the app layer), reason (nullable, bounded), expires_at (nullable, set for a temporary mute or ban), and created_at. Three indexes: (target_participant_id, community_id, created_at desc) for the per-member sanction-state query and the merge lookup by target, (actor_participant_id) for the merge lookup by actor, and (community_id, created_at desc) for community-wide history.
- The moderation action registry (`src/lib/registry/moderation-actions.ts`): the canonical, app-layer set of actions (warn, mute, unmute, kick, ban, unban), a const array, the derived type, and a Zod schema, with no Postgres enum. The route action enum is projected from it, so the OpenAPI spec cannot drift. Its names (MODERATION_ACTIONS, ModerationActionName) are kept distinct from the moderation_actions row type (ModerationAction).
- The moderation action route and service: `POST /api/v1/moderation/actions` (`src/routes/moderation.ts`, `src/services/moderation/actions.ts`), service-token gated, with the registry-derived action and platform enums in the OpenAPI spec. It resolve-or-creates the community, the actor, and the target (reusing the phase 2 paths and their race handling, so a ban can target a member who never posted), then in one transaction appends the log row (expires_at set to now plus durationSeconds for a temporary mute or ban, otherwise null) and applies the membership effect, returning the recorded action and the target's resulting sanction state. Effects: warn logs only; mute or unmute toggle the muted state; kick soft-leaves any membership with no lasting sanction; ban soft-leaves any membership and sets the banned state; unban clears the banned state without rejoining. kick and ban reuse the phase 4a soft-leave, extracted as `deactivateMembershipRow` and shared between the leave route and moderation, run inside the action transaction so the log row and the deactivation commit atomically. Inputs are validated and bounded: an unknown action is a 400, durationSeconds must be a positive integer within a sane bound (ten years), and reason is bounded text.
- The derived sanction state and the two service-token reads (`src/services/moderation/sanction-state.ts`): the state is computed at read from the log and never stored (a member is muted when the most recent mute outranks the most recent unmute and has not expired, and banned by the same rule with ban and unban). One DISTINCT ON query, served by the (target, community, created_at desc) index, reads the latest event of each sanction action. `GET /api/v1/moderation/state` returns the current state (muted, mutedUntil, banned, bannedUntil) for a member resolved without creating, or the cleared state when the member or community is unknown. `GET /api/v1/moderation/history` returns that member's actions as target in the community, newest first, paginated (a 1-based page and a bounded page size) with the response envelope's pagination block.
- moderation_actions joined the centralized participant-owned-relations list (`src/services/participants/owned-relations.ts`) as the first relation with two participant foreign keys: a merge re-points BOTH actor_participant_id and target_participant_id from the ghost to the survivor (a straight update of each, no combine, since a log row collides with nothing), and the USER_HAS_DATA guard now counts a participant as the actor or target of any remaining row. The OwnedRelation contract gained a declared participantColumns set, and the catalog assertion (`test/db/owned-relations-catalog.test.ts`) now cross-checks the registered columns against the Postgres catalog, so a future second column left out of a relocation fails loud, not only a missing table.
- Moderation tests (`test/db/moderation.test.ts`): the migration shape (no updated_at, no trigger, the three indexes), each action and its effect, the derived state across expiry and reversal, a ban on a non-member, the two reads (current state and paginated newest-first history), the merge re-pointing both foreign keys, and the validation cases (unknown action, out-of-bound durationSeconds, over-long reason). The catalog assertion was extended to the two-foreign-key case.
- Network engagement leveling (phase 4b-i), the lifetime network-wide leveling system and the second slice of the engagement core. Experience accrues across the internet of communities on a no-maximum polynomial curve, and a capped contribution value is derived for the phase 5 True Score signal.
- The `network_xp` bigint column on participants (the third migration, `drizzle/migrations/0002_magenta_vision.sql`, a plain append-only ALTER; 0000 and 0001 untouched, and the existing set_updated_at trigger already maintains updated_at). It is a participant's lifetime experience total across all communities, account-wide and uncapped (never per-community in this slice). network_xp is a participant-level scalar, not a participant-owned child relation, so it does not join the owned-relations list and the catalog assertion is unaffected. Drizzle reads it in number mode (every reachable XP is far below 2^53, and the curve is floating-point).
- The leveling module (`src/lib/leveling/curve.ts`), pure derive-at-read functions with no I/O: `xpForLevel` (the total XP to reach a level, `floor((level / XP_SCALE) ^ EXPONENT)`), `levelForXp` (its exact inverse, the closed form corrected against the integer thresholds so the level is right at every boundary, with no maximum), and `trueScoreContribution` (a value in [0, 1] that rises with level and saturates at the cap). Constants: `XP_SCALE` 0.4879, `EXPONENT` 2.15, `CONTRIBUTION_CAP_LEVEL` 50. The level and the contribution are derived from network_xp at read and never stored.
- The engagement route and service: `POST /api/v1/engagement` (service-token gated, the registry-derived platform enum in the OpenAPI spec) and `src/services/engagement/grant.ts`. The call a bot makes on a trackable interaction: it composes the phase 4a ensure-membership, then accrues a flat XP grant (default 20) gated by a per-community Redis cooldown (default 60 seconds). The cooldown is a single ncn:-namespaced key (`engagement:cooldown:{communityId}:{participantId}`) set with SET NX EX: the call that sets the key grants and increments network_xp atomically (a single UPDATE), while a call inside the window is a no-op success (granted false). The cooldown is per community, so a participant accrues from each community independently. This is the first real use of the Redis client wired since phase 1. The response reports the participant (id, network_xp, the derived network level), the membership, granted, leveledUp, and the previous level on a level-up.
- The merge extension: claim-and-merge now folds the ghost's `network_xp` into the surviving identity-bearer (a single atomic increment inside the existing merge transaction) before the ghost is deleted, so the survivor's lifetime total and derived level reflect the combined experience.
- Network leveling tests: unit tests for the pure curve and contribution (`test/leveling.test.ts`: the inverse exactness and boundary correctness across a wide range, the no-maximum growth, and the contribution saturating at the cap) and DB-and-Redis integration tests for the engagement route (`test/db/engagement.test.ts`: the first grant and its level-up, the across-threshold level-up, the cooldown no-op, cross-community independence, and the two-simultaneous-engagement SET NX gate granting exactly once), plus a merge-sum case in `test/db/claim-and-merge.test.ts`.
- The community membership lifecycle (phase 4a), the first write path for community_members and the first slice of the engagement core. Two service-token routes under `/api/v1`: `POST /api/v1/memberships/ensure` (resolve-or-create the participant and the community from platform ids, then create, reactivate, or no-op the membership) and `POST /api/v1/memberships/leave` (soft leave). Both compose the phase 2 resolve-or-create paths, return the response envelope, and carry the registry-derived platform enum in the OpenAPI spec under the serviceToken scheme.
- The membership lifecycle service (`src/services/memberships/lifecycle.ts`). ensure-membership is idempotent and race-safe: it resolves the participant (a ghost if new) and the community outside the membership transaction (so each resolve's own unique-violation re-resolve still works), then ensures one membership row inside a transaction with a FOR UPDATE lock and a retry on the (community_id, participant_id) unique violation. A first call creates the row (active, permission_level 0); a prior leave reactivates it in place (preserving created_at and permission_level); an active membership is a true no-op (no write, updated_at untouched). leave resolves the participant and community without creating them and marks an active membership inactive (active false, left_at now); a missing participant, community, or membership, or an already-inactive one, is an idempotent no-op success.
- The `active` boolean (not null, default true) and `left_at` timestamptz (nullable) columns on community_members, added by the second migration (`drizzle/migrations/0001_narrow_jackpot.sql`), a plain append-only ALTER (0000 untouched; the existing set_updated_at trigger already maintains updated_at). active is the soft-leave flag; left_at records when a membership last went inactive and is null while active.
- community_members joined the centralized participant-owned-relations list (`src/services/participants/owned-relations.ts`) with a re-point-or-combine merge relocation. Where the survivor is not a member of the community, the ghost's membership row is re-pointed; where both are members (a community spanning two platforms), the rows combine into the survivor's: permission_level becomes the higher of the two, active becomes true if either is active, left_at becomes null if the combined membership is active else the later of the two values, the survivor's created_at is unchanged, and the ghost's row is deleted. The USER_HAS_DATA guard now covers memberships, so a merge that left one behind would abort. This closes the phase 3 deferral that kept community_members out of the list.
- The owned-relations catalog assertion (`test/db/owned-relations-catalog.test.ts`): a test that introspects the Postgres catalog for every table with a foreign key to participants and asserts each is registered in the participant-owned-relations list. With platform_accounts and community_members both registered, it passes; a future participant-owned table added without registration fails this test rather than silently dropping rows on a merge. Kept a test, not a boot-time check, so the lazy-connection invariant holds.
- Membership lifecycle integration tests (`test/db/membership-lifecycle.test.ts`): ensure create and idempotency, rejoin reactivation preserving created_at and permission_level, leave and its no-op cases, the merge re-point and both combine branches (active-if-either and the later left_at), the guard relocating an account and a membership together, and the two-simultaneous-ensure concurrency convergence.
- Verification-driven claim-and-merge: `POST /api/v1/participants/claim` (service-token gated), accepting a verification assertion (platform, platformUserId, noclulabsIdentityId) and attaching a platform account to its noclulabs identity. This is the mechanism that unifies cross-platform ghosts into one identity-linked participant. Represented in the OpenAPI spec with the registry-derived platform enum, under the serviceToken security scheme.
- The claim-and-merge logic (`src/services/participants/claim.ts`), one transaction with five cases: resolve-or-create the account if unseen (reusing the phase 2 path); already linked (idempotent, ensure verified); linked to a different identity (409, change nothing); claim a ghost in place (set the identity, verify the account); and merge a ghost into the participant that already holds the identity (relocate the ghost's owned rows, verify the claimed account, delete the ghost). The survivor of a merge is always the identity-bearer; the response reports the outcome (claimed, already_linked, or merged) and the removed ghost id on a merge.
- The USER_HAS_DATA guard and a single centralized list of participant-owned relations (`src/services/participants/owned-relations.ts`). The relocation step and the guard iterate the same list, so they cannot drift: the guard aborts the merge (rolling back) if the loser still owns any row after relocation. In this phase the only owned relation is platform_accounts; phase 4 extends the list for community memberships and XP.
- Concurrency handling for claim-and-merge: each claim locks the platform account row (the stable anchor that survives every case) and the touched participant rows in id order with SELECT ... FOR UPDATE, and retries on a lost claim-in-place race (the noclulabs_identity_id unique violation) or a transient serialization failure or deadlock (a new `isRetryableTransactionError` helper alongside `isUniqueViolation`, both walking the Drizzle cause chain). Two simultaneous claims of the same account and identity converge on one survivor with no duplicate participant.
- Claim-and-merge integration tests (`test/db/claim-and-merge.test.ts`): all five cases, the USER_HAS_DATA relocation of a multi-account ghost, the concurrency race, and route validation (uuidv7 identity ids, malformed id, unknown platform, missing token).

### Changed

- The four bibles were reconciled with the deployed reality: the current state updated to deployed and live with the production flag states, and the exposure language corrected from a Caddy-fronted public surface to the current no-public-route shape (all bridge traffic outbound over the private Docker network, and the summon endpoint disabled and unexposed until noCluBot), with the earlier Caddy-fronted host-port-3000 shape reframed as intended future state. The committed `docker-compose.yml` reconciliation, the calendar-semver tagging pass, and the DEPLOYMENT.md hardening items are recorded as follow-ups in ROADMAP.md.
- The noclulabs.com base client (`src/lib/noclulabs/client.ts`) gained `noclulabsGetReadable`, a GET that returns the status and parsed body for a 2xx or for any readable non-2xx (surface C's 422), built over the same authed core as `noclulabsGet` and `noclulabsPost`, so the score client reuses the auth, timeout, and error mapping. `noclulabsGet` (body-only, throws on any non-2xx) is unchanged, so the verify-sync poller is unaffected. The route body parser (`src/routes/parse.ts`, `parseOrThrow`) gained an optional status-code parameter defaulting to 400, so every existing caller is unchanged and only the summon route passes 422.
- The noclulabs.com base client (`src/lib/noclulabs/client.ts`) was refactored to share one authed core between a GET and a new POST, so the emit client reuses the bearer attachment, the AbortController timeout, the typed error mapping, and the token-never-logged discipline rather than duplicating them. `noclulabsGet` keeps its exact prior behavior (the poller is unaffected); `noclulabsPost` adds a JSON body and an optional set of readable non-2xx statuses, so a caller can inspect a meaningful body (surface A's 422) while 401, 500, and any other non-2xx still map to a thrown `NoclulabsClientError`.
- The verify-sync scheduler plugin (`src/plugins/verify-sync.ts`) now drives the shared claim-and-emit wrapper instead of the bare claim, so a claim the poller drives emits the resulting participant's contribution post-commit, exactly like a route-driven claim. The emit is swallowed inside the orchestration, so the poller's page-atomic, failure-no-advance behavior is unchanged (only a claim error can propagate to the poller).
- The dev Postgres host port moved from 5433 to 5439 (docker-compose.dev.yml, the CI Postgres service, the .env.example default, and the test and CI DATABASE_URL values). 5433 collides with the noclulabs.com dev Postgres on the developer machine, so the two could not run side by side. The container port stays 5432.
- The docker-compose.dev.yml Postgres volume now mounts at `/var/lib/postgresql` instead of `/var/lib/postgresql/data`. The postgres:18 image stores data under a major-version subdirectory and rejects the legacy mount path, so local dev Postgres could not start before this change. CI is unaffected (its service Postgres has no volume mount).
- Vitest now runs test files serially (`fileParallelism` false). A second DB-backed test file shares the single test database with the first, and isolation is by truncation between cases, so files must not run concurrently.
- The global rate limiter now exempts the liveness probe (`GET /health`) and the trusted service-token resolve routes per route (`config.rateLimit` false), so probes and bots are never throttled. The limiter stays registered for any future public-facing tier.

### Fixed

- The committed `docker-compose.yml` was aligned with the deployed configuration: the published host port was removed, a dedicated `redis` service was added, the external `noclulabscom_default` network was joined, and configuration is now loaded via `env_file`. The repository's compose file matches production rather than the earlier single-service, Caddy-fronted intent. The alignment was made in PR #13, which merged without its own changelog entry; this entry records it to satisfy the every-PR-gets-an-entry discipline.
- The BigInt-to-Number preSerialization hook recursed into every object, which flattened Date values (for example a row's created_at) to `{}` and broke date-time serialization. It now recurses into plain objects and arrays only, leaving Date and other class instances intact. The bug was latent in the bootstrap (the health route returns no dates) and surfaced with the first timestamp-returning routes.
